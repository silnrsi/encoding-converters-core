AdaptIt Knowledge Base Lookup Converter

This converter allows you to use an AdaptIt Knowledge Base to provide data for a whole-word-form lookup/replacement transduction process. This can be used by client applications that want to access the database of Source to Target word mappings in an existing AdaptIt Knowledge Base or one you create to use without the Adapt It program. It can also be used as part of a complex, lexically-augmented transliteration process (see below).

Unlike most of the other converters (e.g. CC table, TECkit, Perl, etc), this one only replaces whole words or phrases (up to 10 words). If you have an AdaptIt Knowledge Base entry that maps the word end to finish, for example, this converter will not replace the word friend with frifinish (like the other converters will).

If you are using an existing Adapt It knowledgebase, you must use AdaptIt v 3.1 or newer and have it configured to save the knowledge base as an .XML file in order for this EncConverter to be able to read it.

1.     If you are using an existing Adapt It knowledge base, or if you are creating one without the intention of using it in the Adapt It program. Choose the encoding of the data (i.e. Non-Roman/Unicode vs. Legacy/Ansi). Clicking the former will list the projects in the <My Documents>\Adapt It Unicode Work folder, and the latter will list the projects in the <My Documents>\Adapt It Work folder.

2.     Choose the project desired in the Projects list box. This is automatically populated from the projects available on the local machine for the current user.

For an AdaptIt Transliteration Project (see below), the transliteration data will be in the normal project Knowledge Base file; not the Glossing Knowledge Base. However, it is possible to access a Glossing Knowledge Base if desired.

Note that if you access an Adaptation Knowledge Base (i.e. from an AdaptIt project used to adapt texts from one language to another—which most likely will contain ambiguities), then the converter will return a string containing all the ambiguities for the given lookup word in the Ample ambiguity format (i.e. %count%form1%form2%...%) .

For example, if your AdaptIt Knowledge Base has an ambiguity for the word /से/ 'from, with' in the Source language, which sometimes means /ते/ 'from' and sometimes /कन्‍ने/ 'with' in the target language, then if you attempt to process the word /से/ with this converter, it will return the string /%2%ते%कन्‍ने%/.

If you have such values in a document readable by Microsoft Word, then you can use the Word Pick Document Template to simplify disambiguating these tokens.

5.     Click the  button to bring up the Converter Selection dialog if you want to use another converter as a fallback in the case where the word(s) of the string being converted are not in the Adapt It knowledge base (e.g. so you can fall back to a TECkit or other algorithmic approach).

Saving the converter in the System Repository:

By default, if you click the Apply or OK buttons, the configured converter will be returned to the client application as a temporary converter for temporary use. Once the client application (e.g. FieldWorks or Word) is closed or releases the converter, it will no longer be available. If you want the converter to be permanently available to client applications, then you must explicitly add it to the System Repository using the Save in System Repository button.

When you click this button, the following dialog box will be displayed to query for a friendly name by which the converter will be known in client applications:

Complex Lexically-Augmented Transliteration

One typical use of this converter is to provide a lexical augmentation to what is otherwise a simple algorithmic transliteration scheme.

If you have an algorithmic or character-based transliterator which does an insufficient job of transliterating between scripts due to inherent ambiguities between them, you can overcome the limitations by adding a previous (lexical) process to handle the transliteration of exceptions. As of version 3.1, AdaptIt has a new mode which allows you to create a Transliteration project[1] to create, add to, and maintain the database of transliteration exceptions. This database of exceptions can then be used for transliteration by using the AdaptIt Knowledge Base Lookup Converter as follows:

·       The AdaptIt Knowledge Base containing the lexical exceptions is configured as the main converter.

·       The character-based transliterator (e.g. an ICU transliterator, a TECkit map, or a CC Table) would be the Fallback converter that is called if the word(s) of the string being converted are not found in the Adapt It knowledge base. As of SILConverters 5.4, this new feature is available directly in the Setup dialog for the AdaptIt knowledge base converter as shown by the number 6 in Figure 1 above.

Such a system allows you to use AdaptIt for very rapidly creating the database of (lexical) exceptions.

Configuring AdaptIt as a Transliteration Project

The Knowledge Base accessed by this converter type is created by AdaptIt v. 3.1 or newer.

Create the AdaptIt project with the following characteristics:

·       The Source language refers to the source script and the Target language corresponds to the target script.

·       When you get to the step of opening a document, make sure the Save Document and KB as XML box is checked (as it is by default).

·       On the Tools menu, select the SIL Converters... menu item:

Figure 2: AdaptIt Tools Menu

 

Note that these menu items will be grayed out (as here) unless you have EncConverters v 2.2 installed on your system. This utility is available as part of FieldWorks 3.1 or it can be downloaded from this website: http://scripts.sil.org/EncCnvtrs

Selecting this menu item will bring up the Choose SIL Converter dialog box which shows you the EncConverter that will be used to pre-process the Source word before putting it in the Target word box (though, initially this dialog box will be empty):

Figure 3: Choose SIL Converter dialog box

 

Click the ... browse button to bring up the Choose an Existing Converter dialog box from which you can choose the EncConverter to use. Once you've selected the desired converter, click OK to return to this dialog and the information about the selected converter will be shown (as above).

In a lexically-augmented transliteration scheme, the converter you choose at this point would correspond to the character-based transliterator. This way, as you go through a text in AdaptIt, you'll see immediately what the character-based transliterator generates. If the transliteration is not acceptable, then you can type in the correct value, hit the Enter key, and the exception will be added to the AdaptIt Knowledge Base.

·       On the Advanced menu, select the Use Transliteration Mode menu item, which turns on a special navigation mode in AdaptIt:

Figure 4: AdaptIt Advanced menu

 

In the normal adaptation and glossing modes, all words are entered into the Knowledge Base whether they are accepted as is or whether they are re-typed and then accepted. By contrast, in Transliterate mode, you use a special key sequence to step through the text, which determines whether the word is added to the Knowledge Base (as an exception) or not. The key sequences active in Transliterate mode are:

·       As with a normal adaptation project, hitting the Enter key will cause the current word—whether retyped or not—to be added to the Knowledge Base as an exception.

·       To accept the form generated by the character-based transliterator (i.e. generated by the configured SILConverter), type Ctrl+Enter. This key sequence will cause the AdaptIt project to remember the source word—so that subsequent occurrences of it will be automatically skipped over—but also to indicate that it isn't an exception. This null record indicates to AdaptIt that the character-based transliterator is sufficient for this particular word and no exception record is needed.

In this way, you can process a text word-by-word to transliterate it, and wherever the character-based transliterator does not generate the correct form, you can very quickly add the exception to the AdaptIt knowledge base. As the knowledge base is filled over time, you will be able to very rapidly process texts. And with such an AdaptIt project, you can then create a lexically-augmented EncConverter solution by creating a new Primary-Fallback converter with the AdaptIt Knowledge Base Lookup Converter as the Primary, and the algorithmic converter as the Fallback converter.


[1] A Transliteration project is one in which the same language is used for both the Source and Target words, however the Source language refers to the source script and the Target language corresponds to the target script.

 

As such, you normally won't have any ambiguities in the Knowledge Base, except perhaps words that are homonyms in the Source script, but not in the Target script.
